## 3 Uses, 3 Tofus

Given that I knew absolutely nothing about VRChat development at the time,\
the early development process was essentially me walking into wall after wall.

At first, I had no idea that\
custom interaction logic in a VRChat World has to be built with Udon / UdonSharp,\
nor did I understand that Worlds and Avatars use completely different SDKs and feature architectures.

I genuinely believed Avatar stuff and World stuff were interchangeable...

Which means I naively assumed I could bundle all 3 uses —\
Pickup, Sit, and Avatar accessory —\
into one single sea slug Prefab, and let users pick which feature to enable.

Well... ~~I thought wrong.~~

The World version and the Avatar accessory version must be built separately.

And even within a World,\
putting Sit (VRC_Station) and Pickup (VRC_Pickup) on the same object can make them interfere with each other.

So practically speaking,\
giving each sea slug one main function makes it much simpler to use.

In other words, if I wanted all of these features,\
I'd need 2 sets of files and 3 sea slug Prefabs:
* World / Sit (for sitting)
* World / Pickup (for grabbing and throwing)
* Avatar / MA toggle (for wearing)

Just understanding this took me about a full evening of repeated wall-slamming.\
~~I came out of it teary-eyed and thoroughly defeated.~~

So at the time, I honestly thought "Udon" was just a name —\

~~Which is true. But also very much not true.~~

Only when I actually had to write my own feature did I slam face-first into this wall.

My first feature was extremely simple:
* When a player sits down, the tofu plays a squished BlendShape animation
* When the player leaves, the tofu returns to normal

So I naturally did what you'd do in regular Unity:\
created a normal C# script, wrote a very simple `SitSquish.cs`.

Then I started testing, full of hope, and...

Nothing.

Absolutely nothing happened.

wwww... Huh?\
Why? Did I write something wrong? 囧

I checked the code over and over and found nothing.

Only after researching did I learn: a regular Unity C# script\
can't be used directly as an executable script in a VRChat World.\
You have to create a `U# Script` instead.

When you create a U# Script, Unity generates a `.cs` file to write your code in,\
plus a linked `UdonSharp Program Asset`.

The code still lives in the `.cs` file, but VRChat doesn't run that C# directly.\
It has to be converted by UdonSharp first before it can run in a World.

UdonSharp looks like C# and mostly writes like C#,\
but not every C# feature works in it.

It only supports a subset of C#, has its own restrictions,\
and its own dedicated ways of handling VRChat World interactions.

For example, detecting when a player sits down or leaves a seat\
can't be done the regular Unity way — you have to use the events VRChat provides.

On top of that, if a feature needs to be visible to all players,\
you also have to think about object ownership and network sync.

As for the deeper principles behind all this...

Honestly, I still don't fully get them. `(ㆆᴗㆆ)`\
~~I'm really not built for programming.~~

But by this point, one thing was absolutely certain:\
UdonSharp plays by its own rules.

## Feature #1: Sit Down, Get Squished

Once I knew the correct way to create UdonSharp scripts,\
the sit-and-squish feature actually came together without much trouble.

Initially, I recorded the tofu's BlendShape in an Animation Clip\
and used an Animator to play the squish and recovery animations.

If squishing were the only feature, that approach would've been perfectly fine.

But once pose switching entered the picture,\
every sitting pose would need its own Animation Clip too.

You could combine poses and squish via Animator Layers or extra States,\
but that means managing more parameters, transitions, and playback states —\
and the whole structure keeps getting more complicated.

Since the squish effect is really just a change in BlendShape weight,\
I rewrote `SitSquish.cs` to control the BlendShape weight directly in code.

When a player sits, the script gradually raises the BlendShape weight, slowly squishing the tofu.\
When they stand up, it eases the weight back to `0`, restoring the original shape.

The two features coexist in a much simpler way, and everything's easier to manage.

### The Ankle That Broke Inside Unity

Initially, I posed the model in Blender, recorded the pose as a one-frame animation,\
and exported it together with the humanoid skeleton as an FBX for Unity.

Importing the original T-Pose skeleton alone: Humanoid looked perfectly fine.\
But importing the skeleton together with the sitting animation... things started to look very wrong.

I adjusted the rig and pose over and over, but never found a stable fix.

A senior classmate later explained: having correct Humanoid bone mapping\
doesn't guarantee that the skeleton's base pose is also correct.

Unity tries to calibrate imported skeletons into the T-Pose that Humanoid requires —\
and after calibration, the rotations below the knees were visibly wrong...

So even though every bone name and mapping was perfectly fine,\
opening the Humanoid T-Pose revealed twisted lower legs,\
with the ankles bent at angles that can only be described as fractured.

### Round 1: PoseIndex, Stuck at Zero

My first attempt: all 8 poses share a single VRC Station,\

I built 8 pose States in the Animator,\
connected Any State to each of them,\
and let `PoseIndex` decide which pose plays.

Each UI button then used `SendCustomEvent()`\

In theory, once `SitPoseSelector` received a button event,\
it would update `PoseIndex`, and the Animator would switch to the corresponding State.

But apparently, the part of me that refused to give up\

Dream-me suddenly realized:\
Could it be... that VRChat's sitting behavior is fundamentally based on the **VRC Station**?

Maybe what actually needs switching isn't the Animator State at all —\
but the VRC Station itself?

After all, everything I'd been modifying was animation state inside the same single VRC Station.\
No matter how the parameters and States switched, the player never actually changed Stations. Not once.

In other words: I kept trying to swap the pose within the same hole,\
but I never actually moved the player into a different hole.

As the saying goes: one radish, one hole. (One seat per pose.)

So — fine. Let's just make 8 VRC Stations,\
one Station per pose.

When a player selects a different pose, the script first exits them from the current Station,\
then enters them into the Station matching the new pose.

To pull this off, I needed to modify the original SitPoseSelector\
and add a new script, SitPoseStationRelay:
* `SitPoseSelector`: tracks the current pose, receives button input, and moves the player between Stations.
* `SitPoseStationRelay`: attached to every Station, relaying player enter/exit events back to `SitPoseSelector`.

This way, the controller script always knows which Station the player is in,\
and on pose switch, exits the old seat before entering the new one.

The Blend Tree from before wasn't wasted either —\
it still manages the pose animation for each Station.

## Hehe, You Can't See Me ｜ωФ)ｼﾞｰ

The pose-switching feature that squeezed me dry was finally complete.

But I quickly noticed another problem:\
**The menu shows up in camera shots!**

This is a serious problem!

VRChat players and cameras are inseparable — who doesn't take photos?\
Nobody wants a giant pose menu photobombing every shot!

Absolutely unacceptable!!

So I... yes, my old habit kicked in again.\
~~I assumed I could just move the menu to the **UI Layer** and call it a day.~~

To be fair, the direction wasn't wrong:\
menus on the **UI Layer** are indeed excluded from VRChat's camera by default.

But...\
~~you know... sometimes you really don't want to hear this BUT.~~

**Players could no longer interact with the menu either.**

A bit of research revealed:\
in VRChat, the player's interaction raycast ignores the **UI Layer**.

So a menu on the UI Layer:\
players can still see it, VRChat's camera won't photograph it by default,\
but the player's pointer can no longer click any of its buttons.

Can't photograph it, can see it, can't press it... (wry smile)

Well then... the only remaining option I could think of\
was hiding it with a shader.

However, the `Image` used in UI and the text renderer `TextMeshPro`\
use different texture data and shader structures.

`Image` mainly uses Sprite textures,\
while `TextMeshPro` renders text via glyph data in a Font Atlas.

So I had to write two separate shaders:
* One to hide the UI Image
* One to hide the TextMeshPro text

The goal: a menu that displays normally to the player's eyes, with fully working buttons —\
but which never, ever appears in shots taken by VRChat's camera.

Here's the key concept:\
VRChat provides a global shader variable called `_VRChatCameraMode`.

In the player's normal view, its value is `0`.\
But when the frame is being rendered by VRChat's handheld camera or screenshot function, it becomes a different value.

In other words,\
all my shader has to do is check the value of `_VRChatCameraMode`.

When it's no longer `0`,\
the current frame is being rendered by VRChat's camera or screenshot function.

At that point, use `discard` to throw away those pixels — the menu simply never gets drawn into the camera's image.

With this one check, the goal is achieved:\
**visible to players, clickable by players, invisible only to VRChat's camera.**

## Hands-On Testing with VR Players

I mainly test in non-VR mode,\
so I had no way to confirm how everything feels with an actual VR headset on.

So I asked a friend who regularly plays in VR to help with hands-on testing.\
My friend is an absolute legend... `ᕙ(  •̀ ᗜ •́  )ᕗ`

She helped test:

* Whether the menu and the "…" button are easy to operate in VR

Turns out it was far less intrusive than I'd feared.

### Seat Position Across Different Avatars

Even before my friend's testing,\
I'd already noticed that different Avatars end up sitting in different positions.

Since every Avatar differs in height, proportions, and skeleton,\
some characters float above the cushion, some sink into the sea slug,\
and forward/backward alignment can drift too.

The model I mainly used while creating the poses was Tsuyuna,\
whose original Avatar Height is `0.92m`.

To give Avatars of different body types the best possible experience,\
I tested the poses with roughly 22 Avatars of varying builds.

> For the list of Avatars used in testing, see: [Tools & Materials](https://tsukiyomiyana.github.io/vket-diary/tools-and-materials)

The result: after adjusting their height to `0.90m–0.93m`,\
about 80% of the Avatars could use the current poses without issues.

That said, Avatars in skirts may still experience clipping or deformed hems in certain poses.\
That part, unfortunately, seems to be beyond what I can fully prevent.

~~And I thought that meant I'd found the solution.~~

### Hand the Controls to the Players

During real testing, my friend ran into the same seat-height issue.

She told me that in VR, players can adjust their seated position themselves after sitting down —\
but in non-VR mode, there's no equally intuitive way to do it.

She also mentioned she'd seen height-adjustment levers for seats in other Worlds.

...Huh!? Wait. That's it!!\
Isn't that the answer right there?

YES!!\
**Just let the players adjust it themselves!**

So I went back and added seat position controls to the menu:
* **Seat Height**: adjust the seat's height
* **Seat Depth**: adjust the seat's forward/backward position
* **Reset**: restore the default position

Besides the sliders, I also added `－ / ＋` buttons for fine-tuning,\
so players can dial in the perfect position for their own Avatar.

I'm truly grateful to the friend who did the testing.\
She didn't just provide test videos — she wrote incredibly detailed feedback notes too.
