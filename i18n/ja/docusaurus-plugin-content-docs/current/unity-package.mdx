## 3つの用途、3匹のスティウミ

当時の私は VRChat 開発について本当に何も知らなかったので、\
初期の開発はほぼ壁に激突し続ける日々でした。

そもそも私は、\
VRChat World のカスタムインタラクションは Udon / UdonSharp で作る必要があることも、\
World と Avatar が別々の SDK と機能構造を使っていることも知りませんでした。

Avatar と World のものは共通だとずっと思っていました...

つまり、Pickup、Sit、Avatar アクセサリーの3つの用途を、\
1匹のスティウミの Prefab に全部まとめて、\
プレイヤーに使いたい機能を選んでもらえばいい、と本気で思っていたのです。

結果は...~~考えが甘すぎました~~

World 版と Avatar アクセサリー版は、別々に作る必要があります。

さらに同じ World 内でも、\
Sit（VRC_Station）と Pickup（VRC_Pickup）を同じオブジェクトに載せると、操作が互いに干渉することがあります。

なので実用的には、\
1匹のスティウミに1つのメイン機能、が一番シンプルです。

つまり、これらの機能を全部作るなら、\
2セットのファイルと、3匹のスティウミ Prefab が必要でした：
* World / Sit（座る用）
* World / Pickup（持つ・投げる用）
* Avatar / MA toggle（装着用）

これを理解するだけで、一晩くらい何度も壁にぶつかりました。\
~~涙目になって、すっかり打ちのめされました。~~

~~合ってるけど、合ってない~~

自分で機能を作る段階になって、ようやくこの壁に頭から突っ込みました。

最初に作りたかった機能は、実にシンプルなものでした：
* プレイヤーが座ったら、スティウミが潰れる BlendShape アニメーションを再生
* プレイヤーが離れたら、スティウミが元に戻る

そこで私は、普通の Unity のやり方に従って、\
普通の C# Script を作り、シンプルな `SitSquish.cs` を書きました。

期待に胸を膨らませてテストした結果...

まったく反応なし。

何も起きませんでした。

wwww...え？\
なんで？どこか書き間違えた？囧

何度も見直しても原因が分からず。

調べてようやく知ったのですが、普通の Unity の C# Script は、\
VRChat World の実行スクリプトとしてそのまま使えず、\
`U# Script` を作る必要があるのです。

U# Script を作成すると、Unity はコードを書くための `.cs` ファイルと、\
それに連動する `UdonSharp Program Asset` を生成します。

コード自体は `.cs` に書きますが、VRChat はその C# を直接実行しません。\
UdonSharp による変換を経て、初めて World 内で動作します。

UdonSharp は見た目も書き味も C# に似ていますが、\
すべての C# の書き方がそのまま使えるわけではありません。

C# の機能の一部だけをサポートしており、独自の制限や、\
VRChat World のインタラクションを扱うための専用の仕組みがあります。

例えば、プレイヤーがいつ座って、いつ席を離れたかを判定するには、\
普通の Unity のやり方ではなく、VRChat が提供するイベントを使う必要があります。

さらに、他のプレイヤーにも同じ結果を見せたい機能の場合、\
オブジェクトの所有権（Ownership）と同期の問題も考える必要があります。

より深い原理については...

正直、私もあまり分かっていません `(ㆆᴗㆆ)`\
~~本当にプログラミングの才能がないんです。~~

でも少なくともこの時点で、これだけは確信しました：\
UdonSharp には UdonSharp のルールがある。

## 機能その1 : 座ったらぺちゃんこ

UdonSharp の正しい作り方が分かってからは、\
座ったら潰れる機能自体は、むしろ大きな問題なく完成しました。

最初は、スティウミの BlendShape を Animation Clip に記録し、\
Animator で潰れる・戻るアニメーションを再生していました。

潰れる機能だけなら、このやり方で何の問題もありません。

ただ、後からポーズ切り替えの追加を考え始めると、\
各ポーズにもそれぞれ Animation Clip が必要になります。

Animator Layer や State を増やしてポーズと潰れ効果を組み合わせることもできますが、\
その分、パラメータ、Transition、再生状態の管理が増えて、\
構造全体がどんどん複雑になっていきます。

潰れ効果自体は BlendShape のウェイトを変えるだけなので、\
`SitSquish.cs` を書き直して、プログラムが直接 BlendShape ウェイトを制御するようにしました。

プレイヤーが座ると、プログラムが徐々にウェイトを上げて、スティウミがゆっくり潰れる。\
プレイヤーが立ち上がると、ウェイトを徐々に `0` に戻して、元の形に戻る。

こうすることで、ポーズは Animation Clip に任せ、\
スティウミの潰れ効果はプログラムが独立して制御。

2つの機能がシンプルな形で共存でき、管理もずっと楽になりました。

Blend Tree はパラメータの値に応じて、複数の Animation Clip から選択・ブレンドできます。\
そこで Float パラメータ `PoseBlend` を使い、異なる値を8つのポーズに対応させました。

### 終陣 : 一つの穴に一本の大根

何度挑戦しても、私はずっと同じ場所で足止めされていました。\
同じ設定を見つめすぎて、一種のゲシュタルト崩壊のような状態に陥り、\
見れば見るほど、問題が見えなくなっていました。

そこで一旦プロジェクトを閉じ、ボロボロの心を抱えてベッドに倒れ込み、そのまま意識を失いました。

でも、諦めきれなかった私は、\
~~どうやら夢の中でも挑戦を続けていたようです。~~

夢の中の私は、突然気づきました：\
もしかして...VRChat の座り判定って、「VRC Station」が基準なのでは？

本当に切り替えるべきは Animator の State ではなく、\
VRC Station そのものなのでは？

考えてみれば、私がずっといじっていたのは、同じ1つの VRC Station の中のアニメーション状態だけ。\
パラメータや State をどう切り替えようと、プレイヤーは最初から最後まで、一度も Station を移動していません。

つまり私は、同じ穴の中でポーズだけを差し替えようとしていて、\
プレイヤーを別の穴に移すことを、一度もしていなかったのです。

要するに、「一つの穴に一本の大根」。（1ポーズにつき1つの席。）

それなら、いっそ VRC Station を8つ用意して、\
それぞれの VRC Station に1つのポーズを対応させればいい。

プレイヤーが別のポーズを選んだら、プログラムがまず現在の Station から退出させ、\
それから新しいポーズに対応する Station に入れる。

この切り替え方式を実現するため、元の SitPoseSelector を修正し、\
新しく SitPoseStationRelay を追加しました：
* `SitPoseSelector` : 現在のポーズを記録し、ボタン操作を受け取り、プレイヤーの Station 移動を制御する。
* `SitPoseStationRelay` : 各 Station に付けて、プレイヤーの入退出イベントを `SitPoseSelector` に伝える。

これで、制御スクリプトはプレイヤーが今どの Station に座っているかを把握でき、\
ポーズ切り替え時に、先に元の席から退出して、新しい Station に入れるようになります。

先に作った Blend Tree も無駄にはならず、\
引き続き各 Station が使うポーズアニメーションの管理に使えました。

## えへへ、見えないでしょ ｜ωФ)ｼﾞｰ

私を搾りカスにしたポーズ切り替え機能が、ついに完成しました。

でもすぐに、別の問題に気づきました：\
**このメニュー、カメラに写り込む！**

これは大問題です！

VRChat プレイヤーで、カメラを使わない人なんている？\
座りポーズメニューが丸ごと写真に写り込むのを喜ぶ人は、いないはず！

これは本当にダメ！！

そこで私は...はい、また悪い癖が出ました。\
~~メニューを **UI Layer** に設定すればいいと思い込んだのです。~~

実は方向性としては間違っていません。\
**UI Layer** に設定したメニューは、デフォルトで VRChat のカメラに写りません。

ただし...\
~~ほら...たまに聞きたくない「BUT」ってあるじゃないですか。~~

**プレイヤーもメニューを操作できなくなりました。**

調べてみると、\
VRChat のプレイ中、プレイヤーの操作レイキャストは **UI Layer** を無視するのです。

つまり UI Layer に設定したメニューは、\
プレイヤーには見えるし、VRChat のカメラにもデフォルトで写らないけれど、\
プレイヤーのポインターも、その上のボタンを選択できなくなります。

写らない、見える、押せない...（苦笑）

それなら...私に思いつく方法は、\
シェーダーで隠すことだけでした。

ただ、UI で使われる `Image` と、テキスト表示を担う `TextMeshPro` は、\
使用するテクスチャデータとシェーダー構造が異なります。

`Image` は主に Sprite テクスチャを使い、\
`TextMeshPro` は Font Atlas 内のグリフデータからテキストを表示します。

そのため、2つのシェーダーを別々に書く必要がありました：
* UI Image を隠すシェーダー
* TextMeshPro のテキストを隠すシェーダー

目標は、シェーダーを適用したメニューが、\
プレイヤーの目には普通に表示され、ボタンも普通に操作できて、\
VRChat のカメラで撮影したときだけ、絶対に写らないこと。

ここで重要な概念が登場します。\
VRChat には `_VRChatCameraMode` というシェーダーのグローバル変数があります。

プレイヤーの通常視点では、この値は `0`。\
しかし VRChat のハンドカメラやスクリーンショット機能でレンダリングされるときは、別の値になります。

つまり、\
シェーダーで `_VRChatCameraMode` の値をチェックするだけでいいのです。

値が `0` でなくなったら、\
その画面は VRChat のカメラかスクリーンショット機能によってレンダリングされているということ。

そのときに `discard` でピクセルを破棄すれば、メニューはカメラの画面に描画されません。

この判定によって、\
**プレイヤーには見える、押せる、VRChat のカメラにだけ写らない**という状態を実現できました。

### アバターごとの座り位置

友人に実機テストをお願いする前から、\
アバターによって座った後の位置に差が出ることには気づいていました。

アバターごとに身長、体の比率、骨格が違うため、\
クッションの上に浮くキャラもいれば、スティウミに埋まるキャラもいて、\
前後の位置がずれることもあります。

ポーズ制作で主に使ったモデルはツユナで、\
Avatar Height の元の設定は `0.92m` です。

でも、できるだけ多くの体型のアバターに快適に使ってもらえるように、\
体型の異なる約22種類のアバターで座りポーズをテストしました。

> テストに使用したアバターのリストは、こちらを参照：[使用ツールと素材](https://tsukiyomiyana.github.io/vket-diary/tools-and-materials)

結果、身長を `0.90m～0.93m` に調整すれば、\
約8割のアバターが現在のポーズを問題なく使えることが分かりました。

ただ、スカートのアバターは、ポーズによって貫通や裾の変形が起きることがあります。\
この部分は、私が完全に防げる範囲ではなさそうです。

~~これで解決策を見つけたと思っていました。~~

### 調整はプレイヤーに任せよう

友人の実機テストでも、同じ座面の高さの問題が出ました。

彼女によると、VR 機器なら座った後に自分で座り位置を調整できるけれど、\
non-VR モードには、同じくらい直感的で便利な調整方法がないとのこと。

さらに、他のワールドで座面の高さを調整できるレバーを見たことがある、とも教えてくれました。

...ん！？なるほど！！\

そうだ！！\
**プレイヤー自身に調整してもらえばいいんだ！**

というわけで、メニューに座り位置の調整機能を追加しました：
* **Seat Height** : 座面の高さを調整
* **Seat Depth** : 座面の前後位置を調整
* **Reset** : デフォルト位置に戻す

スライダーに加えて、微調整しやすい `－ / ＋` ボタンも追加して、\
プレイヤーが自分のアバターに合わせてちょうどいい位置に調整できるようにしました。
