## 3 種用途，3 隻臭豆腐

鑑於我當時對 VRChat 開發是真的一竅不通、\
前期的開發過程幾乎可以說是一路撞牆。

我一開始完全不知道、\
VRChat World 裡的自訂互動邏輯需要透過 Udon / UdonSharp 來製作、\
也不理解 World 和 Avatar 使用的其實是不同的 SDK 與功能架構。

我一直以為 Avatar 跟 World 的東西是共通的...

也就是說、我一直天真地以為、\
可以把 Pickup、Sit 和 Avatar 配件 3 種用途、\
全部整合在同一隻臭豆腐的 Prefab 裡、再讓玩家自己選擇要啟用哪種功能。

結果...~~我真是想多了~~

World 的功能和 Avatar 的功能必須分開做。

而即使同樣在 World 裡、\
Sit (VRC_Station) 和 Pickup (VRC_Pickup) 放在同一個物件上也會造成操作互相干擾、\
所以以實際使用來說、讓 1 隻臭豆腐負責 1 個主要功能、操作會比較單純。

也就是說、這些功能如果全都要做、我需要準備 2 套檔案、 3 隻臭豆腐Prefab : 

* World / Sit (坐下用)
* World / Pickup (拿取、丟擲用)
* Avatar / MA toggle (裝在身上用)

光是弄懂這件事、我大概就花了兩個晚上反覆撞牆。\
~~可以說是淚流滿面、灰頭土臉。~~

## 遠看像 C#、近看是 U# (UdonSharp)

這邊要強烈推薦這本 Udon 入門教學 : \
[ぽんぽこほんぽ｜UdonSharp（U#）入門 v4.5｜VRChatワールド・ギミックをゼロから作る初心者向け教科書](https://pon-poko-pon.booth.pm/items/4409611)

如果你對 VRChat World 製作有興趣、我真的非常推薦你看！\
~~它唯一的缺點、就只是長達1152頁的日文而已！~~

進入正題、這是我撞上的第一面牆。

之前準備 Vket 入稿檔案時、\
我只是按照說明、把 `VRC_Pickup` 加到物件上、\
再調整 Inspector 裡的設定、基本就可以直接使用。

因為都是 VRChat SDK 已經準備好的元件、\
完全不需要自己寫程式 (只知道無腦用)。

所以、我當時甚至以為 Udon 只是一個名稱、\
就像是 VRChat 對 Script 的另一種叫法而已。

~~是這樣沒錯、但不是這樣~~

現在要開始自己寫功能時、才終於一頭撞上。

我第一個要製作的功能其實非常單純：
* 玩家坐下時、臭豆腐播放被壓扁的 BlendShape 動畫
* 玩家離開後、臭豆腐恢復原狀

所以我很自然地按照一般 Unity 的做法、\
建立了一個普通的 C# Script、寫了一個很簡單的 `SitSquish.cs`。\
然後滿懷期待地測試、結果...完全沒有反應。

什麼都沒發生。

wwww...欸？\
為什麼？我哪裡寫錯了嗎囧？

反覆檢查了好幾遍都找不到原因、結果查了資料之後才知道、\
原來在 VRChat World 裡、不能直接把普通的 C# Script 掛上去使用、\
而是要建立 `U# Script`。

建立 U# Script 時、Unity 會產生一個用來寫程式的 `.cs` 檔案、\
以及一個與它連動的 `UdonSharp Program Asset`。

程式依然是寫在 `.cs` 裡、但 VRChat 並不會直接執行這份 C#。\
而是要先經過 UdonSharp 轉換、才能在 World 裡運作。

UdonSharp 雖然看起來像 C#、寫起來也很像 C#、\
但並不是所有 C# 的寫法都能直接使用。

它只支援部分 C# 的功能、也有一些自己的限制、\
以及專門用來處理 VRChat World 互動的方式。

例如、要判斷玩家什麼時候坐下或離開座位、\
就不能完全照一般 Unity 的寫法處理、而是要使用 VRChat 提供的事件。

除此之外、如果功能需要讓其他玩家也看到相同的結果、\
還要另外考慮物件的控制權與同步問題。

至於更深入的原理...其實我也沒有很懂 `(ㆆᴗㆆ)`\
~~我真的不是寫程式的料。~~

但至少到了這裡、\
UdonSharp 有自己的一套規則是確定的了。

## 第 1 個功能 : 坐下就壓扁

知道 UdonSharp 的正確建立方式之後、\
坐下壓扁的功能反而沒有遇到太大的問題。

一開始、我是用 Animation Clip 記錄臭豆腐的 BlendShape、\
再透過 Animator 執行壓扁與恢復的動畫。

如果只有壓扁這項功能、這樣做其實沒有什麼問題。

不過、後來考慮追加坐姿切換後、\
每一種坐姿也都需要使用各自的 Animation Clip。

雖然可以透過 Animator Layer 或更多 State 將它們組合在一起、\
但這樣就需要額外管理更多參數、Transition 和播放狀態、\
整體結構也會變得越來越複雜。

由於壓扁效果本身只是改變 BlendShape 的權重、\
只要改寫 `SitSquish.cs`、讓程式直接負責控制 BlendShape 權重。

玩家坐下時、程式逐漸增加 BlendShape 的權重、讓臭豆腐慢慢被壓扁。\
玩家站起來時、再將權重逐漸恢復為 `0`、讓它回到原本的形狀。

## 靈機一動：如果坐下能切換坐姿呢？

做好原本預設的鴨子坐之後、我突然想到：\
「如果坐在上面時、還可以自由選擇不同坐姿、會不會很有趣啊？」

有了坐姿、接下來的問題就是：\
要怎麼讓玩家自由切換這 8 種坐姿呢？

我先把需要的功能列下來：
* 玩家坐下後、要能自由切換 8 種坐姿 → 需要 1 個 UI 選單
* 選單上要能顯示目前選擇的坐姿

### 初陣 : 困在零點的 PoseIndex

第一次嘗試的做法、是讓 8 種坐姿共用同一個 VRC Station、\
透過 Animator 讀取 `PoseIndex` 的數值、切換到對應的坐姿。

我先在 Animator 裡建立了 8 個坐姿 State、\
再從 Any State 分別連到各個 State、\
根據 `PoseIndex` 的數值決定要播放哪一個坐姿。

接著、每個 UI 按鈕都透過 `SendCustomEvent()`、\
呼叫 `_SelectPose0`、`_SelectPose1`……直到 `_SelectPose7`。

照理來說、`SitPoseSelector` 接收到按鈕事件後、\
就會改變 `PoseIndex` 的數值、讓 Animator 切換到對應的坐姿 State。

### 第四陣 : ALL IN Blend Tree!!

也許...問題出在 Animator 的切換結構本身...？

這次、我刪掉原本由多個 State 與 Transition 組成的結構、\
把 8 個坐姿動畫全部放進同一個 Blend Tree 裡。

Blend Tree 可以根據一個參數、在多個 Animation Clip 之間進行選擇或混合。\
因此我使用 Float 參數 `PoseBlend`、讓不同的數值分別對應 8 個坐姿。

真正需要切換的或許根本不是 Animator 的 State、\

畢竟我前面一直在修改的、都只是同一個 VRC Station 裡的動畫狀態。\

## 欸嘿、你看不到我 ｜ωФ)ｼﾞｰ

把我榨得乾乾扁扁的坐姿切換功能總算是完成了。

但我很快就注意到另一個問題：\
**這個選單、會出現在攝影機裡面！**

這可是重大問題！

玩 VRChat 的人、哪有不開相機拍照的？\
應該沒有人會想把一整片坐姿選單、一起拍進照片裡吧！

這樣實在不行啊！！

所以我...對、我又老毛病犯了。\
~~我以為只要把選單設定到 **UI Layer** 就好了。~~

事實上方向也沒錯、\
設定到 **UI Layer** 的選單、確實不會被 VRChat 的相機拍進畫面。

但是...~~你知道...有時候就是不想聽到這個 BUT。~~\
**玩家也無法再與選單互動了。**

查了一下才知道、\
VRChat 遊玩時、是會讓玩家的操作射線忽略 **UI Layer** 的。

因此、設定在 UI Layer 的選單、\
玩家雖然還是看得見、VRChat 的相機也不會將它拍進畫面、\
但玩家的指標、同樣也無法選中上面的按鈕。

拍不到、看得到、也按不到呢...(苦笑)

那...那我能想到的辦法、\
就只剩下用 Shader 把它藏起來了。

不過、UI 裡使用的 `Image`、以及負責顯示文字的 `TextMeshPro`、\
兩者的渲染方式剛好有些不同。

`Image` 是直接讀取 Sprite 貼圖、\
`TextMeshPro` 則需要從 Font Atlas 裡、重新計算文字顯示的範圍。

所以、我得把它們分開寫成了兩個 Shader : 
* 一個負責隱藏 UI Image
* 一個負責隱藏 TextMeshPro 文字

目標是讓套用 Shader 的選單、\
在玩家眼中仍然正常顯示，按鈕也能正常操作、\
但透過 VRChat 相機拍攝時、選單絕不會出現在畫面裡。

這裡就要提到一個關鍵概念、\
VRChat 本身有提供了一個名為 `_VRChatCameraMode` 的 Shader 全域變數。

在玩家視角下、它的數值會是 `0` 、\
但當畫面由 VRChat 的相機或截圖功能渲染時、它就會變成其他數值。

換句話說、我只要讓 Shader 檢查 `_VRChatCameraMode` 的數值即可。\
當它不再是 `0` 時、就代表目前是由 VRChat 的相機進行渲染、\
這時再使用 `discard` 捨棄那些像素、不把選單繪製進相機的畫面裡。

透過這樣的判斷、就能實現 : \
**玩家看得到、按得到、只有 VRChat 的相機看不到**的效果了。

### 不同 Avatar 的座位位置

在請朋友實測前、\
我就已經發現不同 Avatar 坐下後的位置會有所差異。

由於每個 Avatar 的身高、身體比例與骨架各不相同、\
有些角色會浮在坐墊上方、有些則會陷進臭豆腐海兔裡、\
前後位置也可能出現偏差。

我製作坐姿時主要使用的模型是ツユナ、\
其 Avatar Height 原始設定為 `0.92m`。

但為了盡可能讓不同體型的 Avatar 都能有良好的使用體驗、\
我另外使用了約 22 種體型各異的 Avatar 進行坐姿測試。

結果發現、\
約八成的 Avatar 在將身高調整至 `0.90m～0.93m` 後、都能正常使用目前的坐姿。

但裙裝 Avatar 仍可能因為坐姿而出現裙子穿模或裙擺變形、\
這部分似乎就不是我能完全避免的了。

~~原本我以為這就是解法了。~~
