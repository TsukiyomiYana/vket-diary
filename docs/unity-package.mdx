## 3 種用途，3 隻臭豆腐

鑑於我當時對 VRChat 開發是真的一竅不通、\
前期的開發過程、幾乎可以說是一路撞牆。

一開始、我完全不知道、\
VRChat World 裡的自訂互動邏輯是需要透過 Udon / UdonSharp 來製作、\
也不理解 World 和 Avatar 使用的、其實是不同的 SDK 套件與功能架構。

我原本一直以為、Avatar 跟 World 使用的東西是共通的...

也就是說、我曾經很天真地以為、\
可以把 Pickup、Sit 和 Avatar 配件這 3 種用途、\
全部整合在同一隻臭豆腐的 Prefab 裡、再讓玩家自己選擇要啟用哪一種功能。

結果...~~我真是想多了~~

World 版和 Avatar 配件版是需要分開製作的。

而即使同樣在 World 裡、\
Sit（VRC_Station）和 Pickup（VRC_Pickup）放在同一個物件上、操作時也可能互相干擾。

所以、以實際使用來說、\
讓 1 隻臭豆腐只負責 1 個主要功能、操作上會比較單純。

也就是說、如果這些功能全部都要做、\
我需要準備 2 套檔案、 3 隻臭豆腐Prefab : 
* World / Sit (坐下用)
* World / Pickup (拿取、丟擲用)
* Avatar / MA toggle (裝在身上用)

光是弄懂這件事、我大概就花了 1 個晚上反覆撞牆。\

直到現在真的要開始自己寫功能、我才終於一頭撞上這面牆。

我第一個要製作的功能、其實非常單純 : 
* 玩家坐下時、臭豆腐播放被壓扁的 BlendShape 動畫
* 玩家離開後、臭豆腐恢復原狀

所以我很自然地按照一般 Unity 的做法、\

然後滿懷期待地開始測試、結果...

什麼都沒發生。

wwww...欸？\
為什麼？我哪裡寫錯了嗎囧？

反覆檢查了好幾遍、還是找不到原因。

查過資料之後才知道、原來一般 Unity 使用的 C# Script、\
不能直接當成 VRChat World 裡的執行腳本使用、\
而是要建立 `U# Script`。

建立 U# Script 時、Unity 會產生一個用來寫程式的 `.cs` 檔案、\
以及一個與它連動的 `UdonSharp Program Asset`。

程式依然是寫在 `.cs` 裡、但 VRChat 並不會直接執行這份 C#。\
而是要先經過 UdonSharp 轉換、才能在 World 裡運作。

UdonSharp 雖然看起來像 C#、寫起來也很像 C#、\
但並不是所有 C# 的寫法都能直接使用。

它只支援部分 C# 的功能、也有一些自己的限制、\
以及專門用來處理 VRChat World 互動的方式。

例如、要判斷玩家什麼時候坐下或離開座位、\
就不能完全照一般 Unity 的寫法處理、而是要使用 VRChat 提供的事件。

除此之外、如果功能需要讓其他玩家也看到相同的結果、\
還要另外考慮物件的控制權與同步問題。

至於更深入的原理...

其實我也沒有很懂 `(ㆆᴗㆆ)`\
~~我真的不是寫程式的料。~~

但至少到了這裡、我非常確定 : \
UdonSharp 有它自己的一套規則了。

## 第 1 個功能 : 坐下就壓扁

知道 UdonSharp 的正確建立方式之後、\
坐下壓扁的功能反而沒有遇到太大的問題。

一開始、我是用 Animation Clip 記錄臭豆腐的 BlendShape、\
再透過 Animator 執行壓扁與恢復的動畫。

如果只有壓扁這項功能、這樣做其實沒有什麼問題。

不過、後來考慮追加坐姿切換後、\
每一種坐姿也都需要使用各自的 Animation Clip。

雖然可以透過 Animator Layer 或更多 State 將坐姿與壓扁效果組合在一起、\
但這樣就需要額外管理更多參數、Transition 和播放狀態、\
整體結構也會變得越來越複雜。

由於壓扁效果本身、只是改變 Blend Shape 的權重、\
因此、只要改寫 `SitSquish.cs`、讓程式直接負責控制 BlendShape 權重。

玩家坐下時、程式逐漸增加 BlendShape 的權重、讓臭豆腐慢慢被壓扁。\
玩家站起來時、再將權重逐漸恢復為 `0`、讓它回到原本的形狀。

兩個功能就能以比較單純的方式共存、管理起來也方便許多。

照理來說、`SitPoseSelector` 接收到按鈕事件後、\
就會改變 `PoseIndex` 的數值、讓 Animator 切換到對應的坐姿 State。

因此、我使用 Float 參數 `PoseBlend`、讓不同的數值分別對應 8 個坐姿。

不管參數和 State 怎麼切換、玩家實際上從頭到尾、都沒有換過 Station。

也就是說、我一直在想辦法替同一個坑更換坐姿、\
卻從來沒有真的把玩家移到另一個坑裡。

簡單來說、就是「一個蘿蔔一個坑」。

既然如此、那就乾脆準備 8 個 VRC Station、\
讓每一個 VRC Station 分別對應一種坐姿。

當玩家選擇其他姿勢時、程式會先讓玩家離開目前的 Station、\
然後再進入另一個對應坐姿的 Station。

為了完成這套切換方式、我需要修改原本的 SitPoseSelector、\
並另外新增一支 SitPoseStationRelay
* `SitPoseSelector` : 負責記錄目前的坐姿、接收按鈕操作、並控制玩家切換到另一個 Station。
* `SitPoseStationRelay` : 掛在每一個 Station 上、將玩家進入或離開的事件傳回 `SitPoseSelector`。

這樣一來、控制 Script 就能知道玩家目前坐在哪一個 Station、\
並在切換姿勢時、先退出原本的位置、再進入到新的 Station。

先前做好的 Blend Tree 也沒有白做、\
仍然可以繼續用來管理各個 Station 使用的坐姿動畫。

但是...\
~~你知道...有時候就是不想聽到這個 BUT。~~

**玩家也無法再與選單互動了。**

查了一下才知道、\
VRChat 遊玩時、玩家的操作射線會是忽略 **UI Layer** 的。

因此、設定在 UI Layer 的選單、\
玩家雖然還是看得見、VRChat 的相機預設也不會將它拍進畫面、\
但玩家的指標、同樣也無法選中上面的按鈕。

拍不到、看得到、也按不到呢...(苦笑)

那...那我能想到的辦法、\
就只剩下用 Shader 把它藏起來了。

不過、UI 裡使用的 `Image`、以及負責顯示文字的 `TextMeshPro`、\
兩者使用的貼圖資料與 Shader 結構並不相同。

`Image` 主要使用 Sprite 貼圖、\
`TextMeshPro` 則會透過 Font Atlas 裡的字形資料顯示文字。

所以、我得分別替它們寫兩個 Shader : 
* 一個負責隱藏 UI Image
* 一個負責隱藏 TextMeshPro 文字

目標是讓套用 Shader 的選單、\
在玩家眼中仍然正常顯示，按鈕也能正常操作、\
但透過 VRChat 相機拍攝時、選單絕不會出現在畫面裡。

這裡就要提到一個關鍵概念、\
VRChat 本身有提供了一個名為 `_VRChatCameraMode` 的 Shader 全域變數。

在玩家平常的視角下、它的數值會是 `0` 、\
但當畫面由 VRChat 的手持相機或截圖功能渲染時、它就會變成其他數值。

換句話說、\
我只要讓 Shader 檢查 `_VRChatCameraMode` 的數值即可。

當它不再是 `0` 時、\
就代表目前的畫面是由 VRChat 相機或截圖功能進行渲染。

這時再使用 `discard` 捨棄那些像素、不把選單繪製進相機的畫面裡。

透過這樣的判斷、就能實現 : \
**玩家看得到、按得到、只有 VRChat 的相機看不到**的效果了。

### 不同 Avatar 的座位位置

在請朋友實測之前、\
我就已經發現、不同 Avatar 坐下後的位置會有所差異。

由於每個 Avatar 的身高、身體比例與骨架各不相同、\
有些角色會浮在坐墊上方、有些則會陷進臭豆腐海兔裡、\
前後位置也可能出現偏差。

我製作坐姿時主要使用的模型是ツユナ、\
其 Avatar Height 原始設定為 `0.92m`。

但為了盡可能讓不同體型的 Avatar 都能有良好的使用體驗、\
我另外使用了約 22 種體型各異的 Avatar 進行坐姿測試。

> 測試時使用的 Avatar List，請參考：[使用工具與素材](https://tsukiyomiyana.github.io/vket-diary/tools-and-materials)

結果發現、約八成的 Avatar 在將身高調整至 `0.90m～0.93m` 後、\
都能正常使用目前的坐姿。

不過、裙裝 Avatar 仍可能因為坐姿而出現裙子穿模或裙擺變形的情況。\
這部分、似乎就不是我能完全避免的了。

* **Seat Height** : 調整座位高度
